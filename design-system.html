04 generative base · 12 hues × 10 stops · 120 derivations

◇Tonal scales

Every non-neutral color in the desktop app derives from one of these 12 hues. Each hue gets a 10-stop tonal scale (50 through 900) — backgrounds at the low end, fills + borders in the middle, text + emphasis at the high end. 500 is always the canonical wheel value. Liminal-named aliases (-faint, -bg, -tint, -soft, -edge, -glow, -strong) point to specific stops · component CSS reads either form, both resolve to the same hex.

Pigment-aware: opaque hues use linear scales, luminous hues keep saturation in mid-stops (the glow), translucent hues get richer wash variants. Click any stop to jump to its row in the editor.

05 functional bindings · UI surfaces named by purpose, not by hue

◇What each surface uses

Component CSS addresses surfaces by purpose: a primary button is --ui-action-primary, an error banner is --ui-state-error, a live indicator is --ui-status-live. Every binding routes back to a wheel hue at a specific stop. Rebind once · every component using that binding retunes.

06 gradient grammar · adjacent-hue blends · the brand spectrum

◇Gradient grammar

Adjacent on the wheel = allowed (smooth pigment migration). Diametric blends are reserved for specific brand moments. Random cross-wheel blends are not in the system. The brand spectrum (gold → orange → pink → magenta → violet) spans the brand 3 plus their adjacent ornaments — used only for state-rendered ambient orbs/washes per Brand OS Canon v4 Layer C.

brand spectrum · the only diametric blend

--brand-spectrum · used only for state-rendered ambient surfaces (orbs, mist, dreamlike abstraction). Per Brand OS Canon v4: never decorative · derived from measured data, constrained in scope, subordinate to structural signal.

⊘ Do not use the brand spectrum for
decorative page background ✗ no measured-data source · ✗ violates Layer C constraint
button fill / CTA gradient ✗ structural signal must remain · ✗ use --ui-action-primary instead
loading bars / progress ✗ progress is functional state · ✗ use --signal-500 + scale instead

Per Brand OS Canon v4: high-chroma gradients are conditional, never decorative. They appear only when derived from measured data, constrained in scope, subordinate to structural signal.

◇Accessibility

Every text/background pair the desktop client will ship against, scored against WCAG 2.1 AA (4.5:1 for body, 3:1 for large text + UI components). Plus the focus-ring spec, the motion-reduce contract, and the contrast tokens that govern overrides. This section is the engineering hand-off.

Motion · reduced-motion contract

Every animation in the system disables under prefers-reduced-motion: reduce. The first-paint choreography (masthead → brand-3 → wheel) holds at opacity: 1 with no transform. Wheel swatch pulse-in is removed. Motion-driven status indicators (live-dot pulse, vault-pill pulse) fall back to static states with the same color/opacity end-frame. 

@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
  }
}

Color-blind safe pairings

Status colors that depend on hue alone fail for users with deuteranopia, protanopia, or tritanopia. Never use color as the only signal. Pair every status color with a glyph (◇ pill, ✓ check, ⌀ refusal) or a text label. The system already does this — banners always carry a glyph, agent cards always carry a state-label.

Contrast tokens · for overrides

The system will ship --theme-high-contrast as a body-class toggle in v1.0+. When set, all -200 through -400 stops are remapped to -500 through -700 for sharper edges. Tracked in v1.0+ shelf.
